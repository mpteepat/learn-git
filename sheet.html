02 · System Architecture
How Everything Connects
Two-layer architecture: high-level ROS on Jetson Orin NX for intelligence, low-level Arduino for real-time motor control.
Software Algorithms Used
🗺️
Hector SLAM

Simultaneous Localization and Mapping without requiring odometry — relies purely on LiDAR scan matching. Creates 2D occupancy grid maps. Works well indoors with clear wall boundaries. Fails outdoors in open, boundary-free spaces.

Uses scan-to-map correlation (Gauss-Newton optimization). Map saved as PNG + YAML for Navigation Stack.

👁️
YOLOv5 Object Detection

"You Only Look Once" v5 — single-pass convolutional neural network that predicts bounding boxes and class probabilities simultaneously. Runs on the RealSense D435i depth feed.

Classifies objects as Static or Dynamic. Uses depth data to measure person distance. Triggers speed reduction: 10m zone → slow, 3m → full stop. CUDA-accelerated via OpenCV GPU build.

🧠
ROS Navigation Stack

move_base node integrates: global costmap (for full-route planning), local costmap (real-time obstacle avoidance), global planner (Dijkstra/A*), and local planner (DWA — Dynamic Window Approach).

Odometry from encoders provides the /odom topic. Goals set via coordinate shortcuts (x, y, theta). AMCL not used — Hector SLAM handles localization directly.

😊
Face Recognition + gTTS

Uses the face_recognition library (dlib-based) to detect and match faces against a stored database. When a match is found within 6m, the system sends the person's name via the /person_name topic.

Human Interaction node receives the name, calls Google TTS to generate audio, plays "Hello [Name]" through the speaker. 10-second cooldown prevents repeated greetings.

04 · Evaluation
Pros & Cons
Honest critical assessment of every major system component.
✅ Strengths
Hector SLAM works well indoors without wheel odometry — scan-matching alone gives usable maps in structured environments with clear walls.
YOLOv5 provides reliable real-time person detection and classification into static/dynamic categories with minimal false positives in testing.
Linear speed reduction (proportional to distance) creates smooth, safe deceleration when approaching people — much better than abrupt stops.
CUDA acceleration delivers ~3× FPS improvement for image processing, making real-time YOLO + face recognition viable on Jetson hardware.
Two-layer architecture (Jetson + Arduino) cleanly separates high-level intelligence from low-level motor control — modular and maintainable.
Encoder odometry is accurate (<2% error) for straight-line motion — reliable distance tracking for corridor-type deliveries.
Human interaction via face recognition + gTTS is a differentiating feature — robots that greet people by name create much better user experience.
Emergency stop via wireless RC remote is a critical safety feature during development — prevents damage in unexpected situations.
LiFePO4 battery chemistry provides stable voltage discharge, BMS protection, and longer life cycle than standard Li-ion for robot applications.
❌ Weaknesses
Turning mechanism is severely flawed — the robot is not a true differential drive, causing large navigation errors (up to 8m) when any turning is required.
Hector SLAM fails completely outdoors — open spaces with no clear wall boundaries produce noisy, unusable maps. No GPS integration available.
FPS even with CUDA is very low (0.37–3.13 FPS) — system is near real-time at best. Any complex scene with multiple people becomes sluggish.
Face recognition accuracy drops beyond 3m and completely fails beyond 6–7m. Dependent on image quality and number of training images per person.
Motor drives excessive current on inclines/sharp turns, causing BMS to cut power and the robot to stall — degraded battery cells exacerbate this.
No outdoor navigation solution — robot is fundamentally limited to pre-mapped indoor environments. Cannot handle elevation changes, gravel, or open plazas.
LiDAR blind spots for low obstacles (curbs, steps below sensor height) — the 2D scan misses anything below the horizontal scan plane.
Glass and mirror surfaces are transparent to LiDAR infrared — causes mapping artifacts where walls appear missing.
No payload/cargo management system — the "delivery" function has no mechanism for package detection, pickup, or drop confirmation.
No multi-floor capability — no elevator integration, no floor-level awareness, limited to flat single-floor environments.

05 · Better Way
Recommended Improvements
Actionable, prioritized upgrades ordered by impact and feasibility.
1. Fix the Mechanical Drive System HIGH PRIORITY

The single biggest failure: chain-drive with only rear wheels in a non-ideal differential configuration causes massive turning errors. Solutions:

• Option A (Preferred): Switch to 4-wheel differential drive with independent motorized front wheels. This allows true point-turn capability and eliminates turning error entirely.

• Option B: Replace front fixed wheels with a single omni-wheel or caster ball that offers zero resistance in any direction, enabling cleaner differential steering.

• Option C: Add a Mecanum wheel system for holonomic motion — the robot can strafe sideways, enabling much more flexible indoor navigation in tight spaces.

Expected improvement: navigation accuracy should jump from "fails on any off-axis target" to <20cm error across any target coordinate.

2. Replace Hector SLAM with Cartographer or RTabMap HIGH PRIORITY

Hector SLAM is purely LiDAR scan-matching with no sensor fusion. Google Cartographer or RTab-Map would provide:

• IMU fusion: Cartographer fuses LiDAR + IMU, dramatically reducing drift on uneven terrain and sharp turns. Critical for outdoor use.

• Loop closure: Corrects accumulated drift when the robot revisits known areas — prevents the map "drifting" during long deployments.

• 3D mapping with RTab-Map: Uses the RealSense D435i (already installed) to create 3D point cloud maps, enabling detection of low obstacles the 2D LiDAR misses (curbs, stairs).

• Outdoor readiness: Add GPS (e.g., UBlox M8N) and fuse with Cartographer for outdoor localization. GNSS provides global reference; local scan-matching handles precision.

3. Upgrade to YOLOv8 or YOLOv11 + Add Tracking MEDIUM PRIORITY

YOLOv5 was released in 2020. Upgrading to YOLOv8/v11 (Ultralytics) provides 15–25% better mAP at comparable speeds, plus native TensorRT export for Jetson optimization.

• Add DeepSORT or ByteTrack: Multi-object tracking assigns persistent IDs to detected people, enabling the robot to follow a specific person across frames instead of re-detecting each frame independently. This also reduces computation — no need to re-run face recognition on every frame.

• TensorRT optimization: Export YOLOv8 to TensorRT FP16 — achieves 5–10× speedup over PyTorch on Jetson. Can push to 10+ FPS easily.

• ArcFace/InsightFace for recognition: Replace dlib face_recognition with ArcFace (SOTA face embedding) — significantly better accuracy at 6m+ range with smaller gallery.

4. Add ROS2 Migration + DDS Communication MEDIUM PRIORITY

ROS1 Noetic reaches end-of-life in May 2025. ROS2 (Humble/Iron) offers significant advantages:

• DDS middleware: Real-time communication guarantees, better for safety-critical motor commands. No single rosmaster point-of-failure.

• Nav2 (Navigation2): ROS2's Navigation Stack is actively developed with better plugin architecture, recoveries, and behavior trees for complex navigation logic.

• Lifecycle nodes: Better startup/shutdown management — critical for a delivery robot that needs to start reliably every time.

• Security: ROS2 supports SROS2 for encrypted communications — relevant if robot handles sensitive delivery data.

5. Build a Payload Management System MEDIUM PRIORITY

Currently this robot has no cargo system — it's "delivery" in name only. A proper system needs:

• Cargo compartment with locking mechanism: QR-code or RFID-based unlock for recipient verification. Simple servo-driven lid or drawer.

• Load sensor: Confirms package pickup and delivery completion. Prevents robot from departing before package is loaded/retrieved.

• Delivery management software: Web/mobile app to assign delivery jobs, track robot location, log delivery history. Integrate with ROS via REST API bridge.

• Multi-stop delivery: Route optimization across multiple delivery points in a pre-mapped environment.

6. Add IMU + Wheel Slip Detection INCREMENTAL

The RealSense D435i already has an integrated IMU (accelerometer + gyroscope). Enable and fuse it:

• Encoder + IMU fusion (EKF): Extended Kalman Filter combining wheel odometry with IMU data dramatically reduces odometry drift, especially on uneven surfaces where wheels slip.

• Wheel slip detection: Compare encoder speed vs IMU acceleration — when they diverge, flag slip and correct. Currently, wheel slip silently corrupts odometry.

• robot_localization package: ROS package implementing EKF/UKF fusion. Drop-in improvement to odometry with minimal code changes.

7. Upgrade Compute and Sensor Suite INCREMENTAL

• Jetson Orin NX → Jetson AGX Orin: 12× more AI performance (275 TOPS vs 35 TOPS). Enables running multiple neural networks simultaneously without FPS degradation.

• Add 3D LiDAR (Ouster OS0/OS1): Replace 2D RPLiDAR with a 3D unit for full-height obstacle detection. Sees low curbs, prone humans, uneven terrain that 2D misses entirely.

• Add UWB (Ultra-Wideband) positioning: Indoor positioning with 10–30cm accuracy independent of LiDAR maps. Works in featureless corridors where SLAM struggles.

• Upgrade battery to larger capacity: Current 12Ah limits runtime. A 30–40Ah pack would support multi-hour operation for real deployment scenarios.

06 · Knowledge Summary
Full Lecture — Everything You Need to Know
A structured walkthrough of all key concepts from this project, explained from fundamentals up.
LESSON 01
Autonomous Mobile Robots (AMR) & Differential Drive Kinematics

An Autonomous Mobile Robot (AMR) must solve three fundamental problems simultaneously: Where am I? (localization), What surrounds me? (perception), and How do I get there? (path planning). This project addresses all three.

The robot uses a Differential Drive configuration — two independently driven rear wheels with passive front wheels. By controlling the speed ratio between left and right wheels, the robot turns. When both wheels spin at equal speed: straight. When one spins faster than the other: curved turn. When they spin in opposite directions: point turn.

The kinematic equations are: v = R(θ̇R + θ̇L)/2 (linear velocity) and ω = R(θ̇R − θ̇L)/2L (angular velocity), where R is wheel radius and L is the half-distance between wheels. These equations are the mathematical foundation of all robot motion control.

⚠️ The project's critical flaw: the mechanical implementation didn't fully realize differential drive, causing the robot to fail all off-axis navigation targets. Perfect kinematics math means nothing if the hardware can't execute it.
LESSON 02
LiDAR — Light Detection and Ranging

LiDAR works by emitting laser pulses and measuring how long they take to return (Time of Flight, ToF). Distance = (speed of light × time) / 2. The RPLiDAR S2E spins 360° and emits thousands of pulses per second, producing a point cloud — a 2D ring of distance measurements at precise angles.

Key specifications of the S2E: range 0.05–30m, scan frequency 10–15 Hz, angular resolution ~0.1°, ethernet UDP communication (IP 192.168.11.102). This is connected to the Jetson via a local network.

LiDAR limitations in this project: Glass and mirrors are transparent to infrared — the laser passes through, producing "holes" in the map where walls should be. Open outdoor spaces with no reflective surfaces within 30m produce noisy, sparse point clouds that SLAM cannot use reliably. Low obstacles (curbs <15cm) below the scan plane are completely invisible.

LESSON 03
SLAM — Simultaneous Localization and Mapping

SLAM solves a chicken-and-egg problem: you need a map to localize yourself, but you need to know your location to build a map. The algorithm solves both simultaneously by maintaining a probability distribution over possible robot positions while incrementally building the map.

Hector SLAM (used in this project) solves this with scan-to-map matching using Gauss-Newton optimization. Each new LiDAR scan is aligned against the existing map to find the robot's most probable position. The map is an occupancy grid: each cell is labeled free (0), occupied (1), or unknown (-1) with centimeter-level resolution.

Why Hector SLAM works indoors: Indoor spaces have abundant features (walls, corners, furniture) within sensor range. Scan-matching converges quickly because each new scan matches well against existing features. No odometry needed — pure LiDAR.

Why Hector SLAM fails outdoors: Open spaces have sparse features. The algorithm cannot converge — small orientation errors accumulate into large position errors. Better alternatives: Google Cartographer (IMU fusion), or augmenting with GPS for global localization.

LESSON 04
ROS — Robot Operating System Architecture

ROS is not an OS — it's a middleware framework that runs on Ubuntu Linux. Its architecture is a distributed computation graph where individual programs (Nodes) communicate by publishing and subscribing to named data channels (Topics).

Key concepts: roscore is the master registry — it maintains a directory of all active nodes and topics. A Publisher Node sends data to a topic; a Subscriber Node receives it. A Service provides request-response communication. Messages are typed data structures (e.g., geometry_msgs/Twist for velocity commands).

Key topics in this project: /scan (LiDAR data), /odom (odometry), /map (SLAM map), /cmd_vel (velocity commands), /object_type_topic (YOLO classification), /person_name (face recognition result).

TF (Transform) system: ROS tracks the geometric relationship between all robot frames — base_link (robot center), laser (LiDAR position), camera (camera position), odom (odometry frame), map (world frame). The Navigation Stack needs all these transforms defined correctly to function.

rosserial: Bridges ROS topics to Arduino via USB serial. The Arduino receives /cmd_vel Twist messages and translates linear/angular velocity into individual wheel PWM signals for the BTS7960 motor drivers.

LESSON 05
ROS Navigation Stack — Path Planning

The Navigation Stack's central node is move_base. When given a goal coordinate, it orchestrates path planning using two costmaps and two planners:

Global Costmap: A static representation of the entire known map, inflated around obstacles (obstacles appear "larger" by the robot's radius, ensuring safe clearance). The Global Planner (Dijkstra or A*) computes an optimal path from current position to goal across this costmap.

Local Costmap: A dynamic, sensor-updated window around the robot showing real-time obstacles (people, moved furniture). The Local Planner (DWA — Dynamic Window Approach) generates velocity commands that follow the global path while avoiding local obstacles.

Odometry from encoders provides /odom — the robot's estimated position relative to its starting point. Goal coordinates are specified in the map frame (from SLAM). The TF tree bridges these frames. This project used keyboard shortcuts to pre-load known destination coordinates, avoiding the need for a UI goal-setting interface.

LESSON 06
YOLO Object Detection & Depth-Aware Stopping

YOLO (You Only Look Once) is a convolutional neural network that performs object detection in a single forward pass. It divides the image into a grid, and each cell predicts bounding boxes and class probabilities simultaneously. This single-pass design makes it dramatically faster than two-stage detectors (R-CNN family).

YOLOv5 classifies objects with confidence scores. In this project, detected objects are labeled Static (furniture, walls) or Dynamic (people, animals). The Intel RealSense D435i adds depth — using structured light and stereo infrared cameras, it measures the distance to each pixel. The bounding box centroid depth gives person-to-robot distance.

Speed control logic: Three zones control behavior. Zone 1 (>10m): full speed from /cmd_vel. Zone 2 (3–10m): speed multiplier = distance/10, creating linear deceleration. Zone 3 (<3m): velocity = 0, robot stops. The modified_cmd_vel node applies this multiplier before sending to rosserial.

Inertia problem: At Zone 3 trigger, the robot has momentum. Heavy robots don't stop instantly — they coast 0.5–1m past the stop command. Solution: trigger stopping earlier (e.g., 4m) to account for inertia, or implement active braking (reverse PWM briefly).

LESSON 07
Face Recognition & Human-Robot Interaction

The face_recognition Python library (built on dlib) uses a deep convolutional neural network to map face images to 128-dimensional embedding vectors. Faces of the same person cluster close together in this space; different people are far apart. Recognition is then a nearest-neighbor search: compare a detected face's embedding to all stored embeddings.

Factors affecting accuracy: Image resolution (lower resolution at distance = less discriminative features), lighting (extreme shadows degrade embeddings), pose variation (frontal face best, profile worst), and training set diversity (more photos from different angles = better generalization).

gTTS (Google Text-to-Speech): Converts text strings to audio MP3 files via the Google Cloud TTS API. The system builds the greeting string "Hello [Name]", calls gTTS, saves the MP3, and plays it through the speaker via a system audio command. The 10-second cooldown prevents the robot from spamming greetings when a person lingers.

Multi-person handling: Processing time scales with people detected — each face requires an embedding computation and gallery comparison. With 2 people, processing takes ~3–3.7 seconds. This latency is problematic for real-time interaction at walking speed.

LESSON 08
CUDA & GPU-Accelerated Computer Vision

A CPU (Central Processing Unit) has 8–32 powerful cores optimized for sequential tasks. A GPU (Graphics Processing Unit) has thousands of smaller cores optimized for massively parallel computation. Image processing is inherently parallel — each pixel can be processed independently, making GPUs ideal.

CUDA (Compute Unified Device Architecture) is NVIDIA's framework for writing GPU-accelerated code. OpenCV compiled with CUDA support offloads image preprocessing (resize, normalize, color conversion) and neural network inference from CPU to GPU.

The Jetson Orin NX 16GB has an integrated NVIDIA Ampere GPU with 1024 CUDA cores and 32 Tensor Cores for AI acceleration. Enabling CUDA in OpenCV required compiling from source — a ~5-6 hour build process. The result: 3× FPS improvement for object detection alone.

Key insight: Even with CUDA, FPS peaks at ~3 FPS in simple scenes and drops to 0.37 FPS with two people + recognition. For a real delivery robot, 5–10 FPS minimum is needed. TensorRT optimization (converting YOLOv5 to optimized engine format) could achieve this on the same hardware.

LESSON 09
Power Systems, Motor Drivers & Safety Architecture

LiFePO4 (Lithium Iron Phosphate) was chosen over standard Li-ion for good reasons: flat discharge curve (stable voltage throughout charge cycle), higher cycle life (~2000+ cycles vs ~500 for Li-ion), much lower fire risk (thermally stable chemistry), and better performance at high discharge rates needed for motors.

The 24V system is stepped down using two switching regulators: XL4015 (5A, 24V→5V) powers Arduino, encoders, and RC receiver. QS-2405 (8A, 24V→12V) powers Jetson, LiDAR, and display. Switching regulators (buck converters) are 90–96% efficient — far better than linear regulators which waste excess voltage as heat.

BTS7960 H-Bridge Motor Driver: Controls each motor with a PWM signal. The H-bridge allows current to flow in either direction through the motor (forward/reverse). Peak current 43A per driver. The issue: degraded battery cells have higher internal resistance, causing voltage to sag under load — this triggers the BMS overcurrent protection and cuts power entirely.

Safety layers: (1) Software emergency stop via modified_cmd_vel zeroing velocity. (2) Hardware emergency stop button on robot body. (3) Wireless RC stop via FS-i6 switch — most practical during testing since operator doesn't need to chase the robot. Safety should always have physical hardware backup independent of software.

LESSON 10
Lessons Learned & Project Takeaways

1. Mechanical precedes software. The navigation algorithm was correctly implemented, but the mechanical turning flaw made it useless for non-straight paths. Always validate hardware capabilities before assuming software can compensate.

2. Sensor selection determines feasibility. Choosing a 2D LiDAR limits the robot to environments with clear vertical walls at sensor height. A 3D LiDAR or multi-sensor fusion (cameras + 2D LiDAR) would have enabled outdoor operation and low-obstacle detection from the start.

3. SLAM algorithm selection is environment-specific. Hector SLAM for an outdoor delivery robot was a mismatch. Research the deployment environment thoroughly before committing to an algorithm.

4. GPU optimization is not optional for real-time vision. 0.22 FPS without CUDA is unusable. CUDA was essential. TensorRT would be the next required step for deployment-grade performance.

5. Human-robot interaction is valuable. Face recognition + greeting by name significantly elevates user experience. For a real delivery robot, this feature differentiates it from a simple cart. The interaction system design (cooldown timer, priority queue for multiple names) was thoughtful.

6. Modularity enables testing. The ROS topic-based architecture allowed each subsystem (SLAM, navigation, YOLO, face recognition) to be tested independently before integration. This saved significant debugging time.

This project successfully demonstrates ROS-based autonomous robotics fundamentals — SLAM, navigation, AI vision, and human interaction — in a unified system. The mechanical limitations and outdoor failures are fixable engineering problems, not conceptual failures. The software architecture is sound.
